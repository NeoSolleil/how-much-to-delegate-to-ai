# AIエージェントは賢いより忠実──だから任せる範囲は人間が決める

結論から書く。AIコーディングエージェントを使っていて最近いちばん考えているのは「どこまで任せるか」だ。AIが賢いかどうかより、こちらがどこに線を引くか。理由は単純で、AIは賢いというより**忠実**だからだ。

そう考えるようになったきっかけの、小さな実験から書く。

## 忠実さを確かめた小さな実験

年齢計算のバグを一つ、わざと仕込んだコードを用意した。月は見ているが日を見ていないので、誕生日が「今月のこれから来る日」だと1歳ズレる、というよくあるやつだ。

```python
from datetime import date


def calculate_age(birthday: date, today: date) -> int:
    """誕生日と今日の日付から満年齢を返す。"""
    age = today.year - birthday.year
    if today.month < birthday.month:  # day を見ていない
        age -= 1
    return age


if __name__ == "__main__":
    # 一見ちゃんと動いているように見えるケース
    print(calculate_age(date(1990, 3, 15), date(2026, 6, 11)))  # 36（正しい）

    # 誕生日が「今月のこれから来る日」のとき
    print(calculate_age(date(1990, 6, 20), date(2026, 6, 11)))  # 36 と出るが、正しくは 35
```

これを「バグを直して」とだけ雑に投げてみた。情報を渡さなければ見当違いの箇所を触るだろう、というのが事前の予想だった。結果は外れて、あっさり正しく直った。

```python
from datetime import date


def calculate_age(birthday: date, today: date) -> int:
    """誕生日と今日の日付から満年齢を返す。"""
    age = today.year - birthday.year
    if (today.month, today.day) < (birthday.month, birthday.day):
        age -= 1
    return age


if __name__ == "__main__":
    # 一見ちゃんと動いているように見えるケース
    print(calculate_age(date(1990, 3, 15), date(2026, 6, 11)))  # 36（正しい）

    # 誕生日が「今月のこれから来る日」のとき
    print(calculate_age(date(1990, 6, 20), date(2026, 6, 11)))  # 36 と出るが、正しくは 35
```

`(today.month, today.day)` のタプル比較で日まで見るようになり、修正そのものは文句なし。ためしに、期待する挙動・実際の挙動・再現手順まで添えて丁寧に投げ直してみたが、出てきたコードは雑なときと完全に同じだった。バグが小さく明確なら、指示の丁寧さで結果は変わらなかったわけだ。

ただ、どちらの場合も直っていない箇所が一つあった。出力例のコメントだ。

```python
print(calculate_age(date(1990, 6, 20), date(2026, 6, 11)))  # 36 と出るが、正しくは 35
```

コードは直って今は 35 が出るのに、コメントは「36 と出る」のまま残っていた。雑でも丁寧でも、ここは同じだった。

理由を考えると、当たり前の話だった。**頼んだのはロジックの修正で、コメントは頼んでいない。** だから触らない。能力が足りないのではなく、指示の範囲をそのまま守っただけだ。

## 忠実さは諸刃

この「言われた範囲をきっちりやる」性質には裏表がある。

頼んだ範囲はやってくれる。一方で、頼んでいない範囲は手つかずになる（さきほどのコメントのように）。そして逆方向もある。「全部いい感じにして」と言えば、AIは"いい感じ"の範囲を自分で広く解釈して、触ってほしくなかった所まで触る。

![AIは指示の範囲を忠実になぞる](fig1_faithful.png)

どちらも同じ性質の表れだ。AIは指示の範囲を忠実になぞる。だから**範囲を決めるのが人間の仕事**になる。賢いAIに丸投げするほど、この「範囲設定」が抜け落ちて事故る。

## どこで線を引くか

自分は「壊れたときの実害」で分けている。

![どこまで任せるか 実害の大きさで線を引く](fig2_delegation.png)

**任せていい：壊れてもすぐ気づく・すぐ戻せる領域**
ローカルでのコード整形、使い捨てスクリプト、ログの集計、テストの下書き。間違っても被害が手元で完結し、git ですぐ戻せる。ここは自動承認寄りで回す。

**確認してから通す：影響が広がる領域**
共通モジュールやユーティリティの変更、複数ファイルにまたがるリファクタ、依存パッケージの更新。一つの変更が遠くまで波及する。差分は必ず目を通してからコミットする。

**任せない／必ず人間が最終判断：取り返しがつかない・外に出る領域**
認証・認可まわり、クレデンシャルや秘密情報を含む処理、本番への反映、外部に公開される成果物。ここは生成させても"そのまま通す"はしない。

線の引き場所は人やチームによるが、軸が「実害の大きさ × 戻しやすさ」なのは共通だと思う。

## これはセキュリティの発想と地続き

書いていて気づいたが、これは普段セキュリティでやっていることと同じ構造だ。

![自走の範囲を絞る](fig3_security.png)

- **最小権限**：必要な範囲だけ許可する。むやみに全許可（`--dangerously-skip-permissions` のような）で走らせない。
- **確認ゲート**：実行の前に人が見る一手を残す。差分レビューや承認プロンプトを潰さない。
- **影響範囲の限定**：いじってよいディレクトリやスコープを区切る。

要は「自走の範囲を絞る」。便利さと引き換えに自走させすぎると、忠実なAIは指示の解釈いっぱいまで走る。そのトレードオフを、なんとなくではなく意識して取りにいく。

## まとめ

AIが優秀になるほど「線を引く」設計が効いてくる。何をどこまで任せ、どこは人間が握るか。これはツールの性能の話ではなく、使う側の判断の話だ。

忠実なAIは、任せれば任せただけやる。だからこそ、任せ方を決めるのはこちらの責任になる。
